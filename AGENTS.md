## Project Rules

- 只读取当前根目录 `AGENTS.md`，忽略子目录 `AGENTS.md`。
- 进入 `agent_rules` 路由或执行工具初始化前，必须先读 `agent_rules/env.md`。
- `agent_rules/env.md` 中的 `operation_scope` 表示默认允许操作的目录；除 `operation_scope` 和明确规则任务涉及的规则目录外，其他目录默认只读，如需修改必须先通知用户并获得允许。
- 普通规则路由顺序：`agent_rules/env.md` -> `agent_rules/common/common.md` -> `agent_rules/project/project.md` -> 命中的技术栈规则；若规则冲突，以 `project.md` 为准。
- 生成或重建 `agent_rules/project/project.md` 是特殊初始化流程：`AGENTS.md` -> `agent_rules/env.md` -> `agent_rules/common/gen_project.md` -> 正式工程真实文件；禁止读取旧 `project.md`，必须先删除旧 `agent_rules/project/project.md`，再生成新 `project.md`，随后按 `gen_project.md` 执行 Serena 初始化 / setup 和 `codegraph init`，并输出实际索引目录。
- 创建或删除文件后，必须在项目根目录执行 `codegraph index` 重新创建索引。
- 只处理用户明确指出的范围；禁止无关重构、格式化、目录清理、额外功能或顺手优化。
- 只进行编码、静态检查、语法检查，以及“修改部分”和“直接引用修改部分”的联动检查；未经明确授权，不执行 Git、构建、安装、打包、部署、数据库写入或生产操作。
- 必须保持最小必要 diff；禁止无语义变化的删后重加、整段重写或等价改动。
- 代码注释只写必要的业务约束、复杂逻辑或非显然取舍；禁止空洞注释。

## Response Style

- 默认把用户提出的内容先归类为“问题”，以回复、解释、判断、建议为主；不要因为用户指出现象、贴报错、给截图或给路径，就自动修改代码。
- 只有用户明确发出执行类指令时，才归类为“任务”并允许修改代码或文件。执行类指令包括但不限于：`帮我弄`、`帮我完成`、`开始`、`修改`、`更改`、`实现`、`执行`、`生成文件`、`写入`、`修复`、`你来改`。
- 如果用户只是问“为什么 / 是否可以 / 是不是 / 哪里 / 哪个表 / 怎么设计 / 给我总结 / 给我 SQL”，默认只回复结论、方案或 SQL 文本，不直接改代码。
- 如果用户意图不清，先按“问题”处理；只有影响较大且无法合理判断时，才向用户确认是否进入修改。
- 回复中只能使用以下固定语义 emoji 标识，不使用其他 emoji：
  - `🚩 完成`：说明已经完成的修改、检查、生成结果；只有确实完成且用户当前不需要执行任何操作时才能使用。
  - `📌 当前`：说明当前结论、当前状态、当前判断；用户问“是什么 / 为什么 / 是否合理”时优先使用。
  - `⏭ 下一步`：仅在确实存在下一步、需要用户确认、需要用户执行或建议用户继续操作时使用；必须明确列出“需要用户做的事情”。
  - `💡 建议`：说明推荐方案、风险提示、取舍理由；不要和事实结论混写。
  - `🧩 使用组件`：说明本次 MCP / Skills 使用情况。
- 对于小问题，先直接给结论。
- 未经用户明确要求，不主动展开排查过程、技能说明、规则路由或跨文件检索结果。
- 只有当用户明确要求“展开”“定位原因”“帮我修改”“彻查”时，才补充详细过程。
- 每次最终回复时，必须追加 `🧩 使用组件` 小节，明确列出本次使用了哪些 MCP、哪些 Skills；分别写明名称、作用、以及本次实际做了什么。若某一类未使用，也必须明确写“未使用”。
- 如果存在下一步、需要用户确认、需要用户执行或建议用户继续操作，最终回复必须包含 `⏭ 下一步` 小节，并明确列出“需要用户做的事情”，不能与普通回复混在一起。
- 如果任务已经完成且用户当前不需要执行任何操作，最终回复必须使用 `🚩 完成` 小节，并直接写“你当前不需要执行任何操作”；此时不强制输出 `⏭ 下一步` 小节。
- 凡是需要用户执行、查看、打开或确认的本地文件，必须使用可点击 Markdown 链接，不能只写纯文本文件路径；链接目标应使用当前机器可访问的绝对路径。
- 若回复包含事实判断，必须放在 `📌 当前`；若回复包含已完成事项且无下一步用户动作，必须放在 `🚩 完成`；若回复包含建议或取舍，必须放在 `💡 建议`。

## Decision Matrix

- 先分类：默认按“问题”处理；只有命中明确执行类指令，才升级为“任务”。问题只回复结论、方案或 SQL，不改文件。
- 再分大小：小 = 单点、低依赖、可直接完成；大 = 多阶段、跨目录 / 跨模块 / 跨技术栈、高风险或不确定性高。
- 小问题：直接回答，默认不进入 `agent_rules`。
- 大问题：进入 `planning-with-files`，读取 `common.md`、`project.md`，再按技术栈补读命中规则。
- 小任务：若涉及代码 / 配置 / 脚本 / 测试 / 构建 / 接口 / 样式或依赖仓库约定，进入普通规则路由；否则可直接处理。
- 大任务：进入 `planning-with-files` 和普通规则路由；跨多个技术栈时读取全部命中规则。

## Coding Behavior Addendum

- 编码前先判断成功标准和验证方式；大任务、高风险任务或影响数据 / 接口 / 构建 / 跨模块行为的任务，先说明关键假设、歧义、取舍和验证方式。
- 如需求存在多种合理解释，且会导致不同代码、数据或接口行为，必须先向用户确认。
- 简单优先：只实现用户明确要求的能力，不添加推测性功能、一次性抽象、未要求配置项或过度防御逻辑。
- 匹配现有代码风格；只触碰直接相关文件；清理仅限本次改动造成的未使用 import、变量、函数、测试数据或临时文件。
- 验证范围限制在修改部分和直接引用修改部分；是否新增测试取决于风险、现有测试体系和用户授权。

## Execution Rules

- 只有进入 `planning-with-files` 时，才在项目根目录维护 `task_plan.md`、`findings.md`、`progress.md`；否则可直接处理或使用简短计划。
- 处理中发现范围扩大、分支增多、信息量变大或跨目录 / 模块 / 技术栈时，必须升级为大问题 / 大任务流程。
- 大任务计划必须细化到可独立执行、验收和恢复的步骤；禁止只写“后端、前端、联调、检查、收尾”这类粗粒度标题。
- 普通规则路由按 `env.md`、`common.md`、`project.md`、技术栈规则执行；技术栈不能确定时只读前三者。
- 规则文件判断优先级：用户明确指定路径或模块 > 即将修改文件所在目录 > 目标文件所属技术栈 > 仓库根目录标记文件。
- 仓库中存在某技术栈，不等于当前任务必须读取该技术栈规则；规则跟随目标文件、模块和直接调用链。
- 只有识别技术栈标记、生成项目规则或排除歧义时，才查看 `agent_rules/common/gen_project.md`。
- 跨多个技术栈时，读取全部命中规则，并检查相互引用的修改点。
- 常见技术栈规则：Android `android.md`；Vue `vue.md`；Python `python.md`；Java `java.md`。
